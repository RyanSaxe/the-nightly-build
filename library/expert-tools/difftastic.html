difftastic compares syntax trees, so re-wrapping stops reading as a rewrite

It parses both files with tree-sitter and searches for the cheapest path between the two trees, which is why formatting noise vanishes and why a large enough diff falls back to plain line diff.

A one-line function grows a third argument, and git diff repaints all of it

A function took two arguments on one line. It now takes three, wrapped one per line for readability. The only real edit is the third argument and the comma that had to follow the second. Here is what git diff makes of it.

Six lines are flagged and two arguments never changed. The diff is reporting the reformatting, because reformatting is what moved the text, and moved text is all a line-oriented comparison can see. The reader has to reconstruct the actual edit by eye, which is the tax an expert pays on every reindent, every re-wrapped call, every function that slid down the file.

Difftastic answers a different question. It parses both versions of the file, compares the two syntax trees, and reports what changed in the structure rather than in the layout.2 The reformatting above does not change the tree, so it does not appear in the result.

The goal of a diff is to find what stayed the same

The useful way to state the task is inside out. A diff is not really a search for what changed. It is a search for the largest part of the file that did not change, because once the unchanged material is pinned down, the change is whatever is left over. Wilfred Hughes, who wrote difftastic, puts it plainly: the goal of diffing is to work out what has not changed.3

Line diff answers that question at the granularity of whole lines. Two lines match or they do not, and a line that was only rewrapped counts as two different lines. Difftastic answers it at the granularity of the syntax. It runs the file through tree-sitter, a parser that turns source text into a concrete syntax tree and is built to reparse quickly as code is edited.4 The arguments myarg1 and myarg2 are the same tree nodes before and after, whatever column they sit in, so the comparison can hold them fixed and spend its attention on the one node that is new.

That reframing is what makes the rest tractable. If the job is to maximize the shared structure, then finding a diff becomes finding the cheapest set of edits that turns one tree into the other, and cheapest is a word a computer can optimize.

difftastic diffs by finding the cheapest path between two trees

Both trees are flattened into nested lists, so every element is either a list or an atom.3 The comparison is then posed as a graph. A vertex is a pair of positions: one position in the left tree and one in the right.5 The start vertex sits at the beginning of both trees; the end vertex sits after the end of both.5 Getting from start to end means walking both trees to their conclusion, and every step is one of three edits, each with a price:

  • advance on the left only, marking a node as removed;
  • advance on the right only, marking a node as added;
  • advance on both at once because the nodes match, which is the cheapest edge of the three.3

Matching costs less than marking a node novel, so any route that keeps shared structure aligned is cheaper than one that does not. A diff is now a shortest path. Difftastic runs Dijkstra's algorithm to find the lowest-cost route from the start vertex to the end, and that route, read back out, is the edit it displays.5 The unchanged arguments cost nothing to align, so they stay quiet; the added argument is the one expensive edge on the cheapest path.

The size of that graph is its price. It is quadratic, O(L \times R), where L and R are the number of elements in the left and right trees.3 Two properties keep that from exploding. Difftastic never materializes the whole graph. Neighbours of a vertex are constructed only as the search reaches them, so a route that is never explored costs no memory.5 And it discards the obviously unchanged runs at the start, middle, and end of the file before the search begins.3

Wired into git as the external diff, it marks two tokens on that same edit

Difftastic is a drop-in for git's own diff. Pointed at the wrapped function from the opening, through git's external-diff hook, it reports the structural edit instead of the textual one. The binary is difft, and the one-off invocation names it as the external differ for a single command:

Two items carry the change: the comma the second argument grew, and the line that adds the third. The reflow that rewrote every line for git diff is unmarked here. To make it the default for a repository, the same setting goes in ~/.gitconfig under a [diff] external = difft line, after which git diff, git show --ext-diff, and git log -p --ext-diff all render structurally. The config route hands difftastic more than the difftool path does, including file-permission and rename information, so it can show more.6 An independent write-up reports the same isolation on a Rust signature. When a parameter changes from String to impl Into<String>, line diff repaints the whole line. Difftastic greens only the inserted impl Into< and its closing bracket, naming the type as the thing that changed.7

The parse it depends on is also where the costs sit

Everything difftastic does well rests on getting a parse, and each cost is a case where the parse is unavailable or too expensive to search. There are three ways it steps back to a line diff, and they are worth keeping distinct.

The three conditions under which difftastic abandons structural diffing and falls back to a line-oriented one.82
Trigger Default What happens
byte limit 1 000 000 A file larger than this is diffed as text before it is ever parsed.
graph limit 3 000 000 If the search crosses this many graph vertices, it gives up mid-diff and finishes as text.
no grammar — A language with no tree-sitter grammar gets a line diff with word highlighting.

The byte limit (DEFAULT_BYTE_LIMIT) and the graph limit (DEFAULT_GRAPH_LIMIT) are constants in the source, tunable through DFT_BYTE_LIMIT and DFT_GRAPH_LIMIT.8 A third constant, the parse-error limit, defaults to zero.8 The grammar case is the one to plan around: coverage is language by language. The project's README says difftastic supports over thirty languages,8 while the manual's current table enumerates sixty-three grammars and formats, and difft --list-languages is the honest answer for any given install.2 A file in something outside that set still diffs, just as text.

Speed is a real cost, scoped narrowly. Hughes states it in the README: difftastic scales relatively poorly on files with a large number of changes, and can use a lot of memory.8 That is the tail the quadratic search punishes, a file rewritten wholesale rather than edited. On review-sized diffs the independent practitioners who use it report no speed trouble. The recurring complaints in community threads are about the side-by-side display being hard to read and about packaging friction from difftastic's Rust toolchain, not about speed.9 The adoption gap those same users name is reach: difftastic is a terminal tool, not yet wired into the editor panes and the pull-request views where much review now happens.1

A tool you put in front of every git diff has to be maintained, and this one is

Structural diff, not line diff, is a change you make to a command you run dozens of times a day, so its upkeep is a fair question to press. The record is active. The latest tagged release is 0.69.0 from 30 April 2026, with 0.70 already in development, following 0.68 in March 2026 and 0.67 in November 2025.8 The most recent commit lands 21 July 2026, and through the summer the repository takes commits several times a week.10 It is largely one maintainer's project, Hughes himself, with the churn of dependency and parser updates around him.10

Robustness is the honest caveat, and it is one the project states about itself: difftastic regularly ships releases that fix crashes.8 Read one way that is a warning; read against the release cadence it is reassurance, since the crashes are being found and fixed on a tight loop rather than accumulating.

Verdict

Set difftastic as git's external diff and it earns its place on one job line diff cannot do: keeping reformatting, rewrapping, and moved code out of the result, so the real edit is what you read.19 The costs are bounded and legible: a language needs a tree-sitter grammar, a very large or wholesale-rewritten file falls back to text, and the display is a terminal view, not an editor pane.8 The assessment would change if the release loop that keeps it in step with tree-sitter and fixes its crashes went quiet.10
